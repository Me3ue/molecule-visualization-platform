# 拓扑文件（Topology）与轨迹文件（Trajectory）数据源及验证说明（论文体例）

## 摘要

本文档面向本项目（Molecule Visualization Platform）中“拓扑文件 + 轨迹文件”双文件加载流程，系统说明数据来源、文件格式、验证数据集选择依据、以及前后端加载机制。项目在实现层面支持基于浏览器的分子结构渲染与轨迹播放，其中以 **GROMACS（GRO+XTC）** 与 **Amber（PRMTOP+RST7）** 为重点验证对象，并兼容多 MODEL 的 PDB 轨迹表示。为保证可重复性，本文给出公开可访问的数据源网址与标准化验证流程。

---

## 1. 本项目中的数据源类型与来源方式

从工程实现看，本项目的数据来源属于“**用户上传型数据源**”，并非固定内置数据库。即：

1. 前端页面由用户上传拓扑文件（Topology）和轨迹文件（Trajectory）；
2. 系统按文件后缀和用户选择格式进入不同解析路径；
3. 必要时调用后端接口进行格式转换（例如 Amber `rst7` 转多 MODEL `pdb`）。

因此，项目的数据来源可分为：

- **结构/拓扑来源**：PDB、GRO、PRMTOP 等；
- **轨迹来源**：XTC、RST7（及同类重启坐标）、多 MODEL PDB 等；
- **公共数据库来源**：RCSB PDB、MD 数据仓库、教程数据集等。

---

## 2. 推荐公开数据源网址（可用于复现实验）

### 2.1 结构拓扑类（PDB）

1. **RCSB Protein Data Bank（官方）**  
   - 主页：https://www.rcsb.org/  
   - 用途：下载标准 `PDB/mmCIF` 结构文件，作为本项目结构展示与拓扑基础输入。

2. **PDB-101（教学资源）**  
   - 主页：https://pdb101.rcsb.org/  
   - 用途：选取教学友好的小中型蛋白结构进行可视化演示。

### 2.2 分子动力学轨迹/拓扑类（GROMACS/Amber）

3. **GROMACS 官方教程数据（以 GRO/XTC 为主）**  
   - 主页：https://www.gromacs.org/Documentation/Tutorials  
   - 用途：获取典型 `*.gro + *.xtc` 组合，验证本项目 NGL 轨迹播放路径。

4. **Amber / AmberTools 官方资源**  
   - 主页：https://ambermd.org/  
   - 用途：获取 `*.prmtop + *.rst7` 或相关重启坐标文件；验证本项目 Amber 路径与后端转换能力。

5. **MDAnalysisData（公开 MD 示例数据集）**  
   - 文档：https://www.mdanalysis.org/MDAnalysisData/  
   - 用途：下载可复用拓扑与轨迹样例（含多种格式组合），用于跨格式兼容性测试。

6. **Zenodo（通用科研数据仓库）**  
   - 主页：https://zenodo.org/  
   - 用途：检索可公开复现的 MD 轨迹数据（建议按关键词如 `GROMACS XTC GRO`、`Amber PRMTOP RST7`）。

## 3. 用于结果验证的数据集选择（论文语气）

为验证系统在不同分子文件生态下的可用性与鲁棒性，建议采用以下三类数据集开展实验：

1. **结构可视化基准集（PDB-only）**：
   选取 RCSB 中若干典型蛋白结构（小分子结合蛋白、酶、膜蛋白片段等），验证模型加载、渲染风格切换、测量功能正确性。

2. **GROMACS 动力学轨迹集（GRO+XTC）**：
   选取 GROMACS 教程或公开仓库中的中等规模轨迹，验证帧数识别、播放控制（首帧/末帧/步进/快进快退）、截图与录屏导出流程。

3. **Amber 双文件集（PRMTOP+RST7）**：
   选取 AmberTools 生态样例，验证 `rst7` 在后端转换为多 MODEL PDB 后的前端连续帧展示；同时验证转换失败时的单帧兜底与错误提示机制。

上述三类数据共同覆盖了本项目的核心声明：
- 多格式输入；
- 双文件拓扑-轨迹协同加载；
- Web 端可交互播放与可解释状态反馈。

## 4. 文件格式对照表（Topology vs Trajectory）

| 类别 | 典型扩展名 | 是否含键信息/拓扑 | 是否含多帧坐标 | 本项目用途 | 备注 |
|---|---|---:|---:|---|---|
| PDB（单结构） | `.pdb` | 部分（可表达连接信息但不完整） | 否 | 结构展示 / 拓扑输入 | 生物结构领域通用文本格式 |
| PDB（多MODEL） | `.pdb` | 同上 | 是 | 轨迹帧播放（3Dmol） | 常作为“构象集合”表示 |
| GRO | `.gro` | 否（偏坐标与盒信息） | 通常单帧 | GROMACS 拓扑/参考结构输入 | 常与 XTC 配对 |
| XTC | `.xtc` | 否 | 是 | GROMACS 轨迹输入（NGL） | 压缩轨迹，体积小 |
| PRMTOP | `.prmtop/.parm7/.top` | 是（力场拓扑参数） | 否 | Amber 拓扑输入 | 与坐标/轨迹文件配套 |
| RST7/INPCRD | `.rst7/.inpcrd/.restrt/.crd` | 否 | 可能单帧或多帧（取决于文件） | Amber 坐标/轨迹输入 | 本项目可后端转 PDB 多MODEL |
| XYZ | `.xyz` | 否 | 可多帧 | 通用轨迹演示 | 简洁但信息较少 |
| SDF/MOL2 | `.sdf/.mol2` | 分子连接信息较丰富 | 可多构象 | 小分子/构象集合展示 | 在教学中常见 |

## 5. PDB 文件内容与格式详解

PDB 为固定列宽文本格式，常见记录类型包括：

- `HEADER`：结构总体信息；
- `TITLE`：标题；
- `REMARK`：注释；
- `ATOM` / `HETATM`：原子坐标核心记录；
- `CONECT`：连接关系（可选）；
- `MODEL` / `ENDMDL`：多模型边界（用于多帧）；
- `END`：文件结束。

### 5.1 `ATOM/HETATM` 关键字段（概念）

- 原子序号（serial）
- 原子名（name）
- 残基名（resName）
- 链 ID（chainID）
- 残基序号（resSeq）
- 三维坐标（x, y, z）
- 占有率（occupancy）
- B 因子（tempFactor）
- 元素符号（element）

### 5.2 多 MODEL PDB 与轨迹关系

当文件包含多个 `MODEL ... ENDMDL` 片段时，可视为“按帧组织的坐标集合”。
本项目中可将其作为轨迹数据源进行帧播放（尤其在 3Dmol 路径中）。

## 6. Trajectory 文件内容与格式详解

### 6.1 XTC（GROMACS）

- 二进制压缩轨迹；
- 主要存储每帧原子坐标（及时间/盒信息）；
- 不含完整拓扑，需配合 `GRO` 或其他结构文件；
- 优点：体积小，适合长轨迹在线传输。

### 6.2 RST7 / INPCRD（Amber 生态）

- 常用作重启坐标或坐标容器；
- 与 `PRMTOP` 配对提供原子/拓扑语义；
- 在不同工具链中对“多帧”支持存在差异，因此本项目加入后端转换兜底。

### 6.3 多 MODEL PDB 作为轨迹容器

- 文本格式、可读性强；
- 适合教学和调试；
- 文件体积通常大于二进制轨迹（XTC/DCD 等）。

## 7. 本项目中的加载机制（与代码实现一致）

### 7.1 前端加载策略

1. 用户选择拓扑格式与轨迹格式，并上传对应文件；
2. 系统判断进入：
   - **NGL 路径**：GRO+XTC 或 PRMTOP+RST7（推荐真实动力学轨迹）；
   - **3Dmol 路径**：PDB/XYZ/SDF/MOL2 等文本轨迹或构象集合；
3. 计算可用帧数，启用播放控件（首末帧、步进、快进、FPS）。

### 7.2 Amber `rst7` 后端转换机制

- 当检测到 Amber 组合时，前端调用 `/api/trajectory/convert-rst7`；
- 后端基于 `cpptraj` 执行：
  - `parm <topology.prmtop>`
  - `trajin <trajectory.rst7>`
  - `trajout converted.pdb pdb multi`
- 返回多 MODEL PDB 给前端进行帧播放；
- 若转换失败，系统返回错误详情并进行兜底提示。

### 7.3 兼容性建议

- 真实长轨迹优先：`GRO + XTC`；
- Amber 轨迹建议确保服务器已安装 AmberTools/cpptraj；
- 教学演示可选用多 MODEL PDB 以提升可解释性。

## 9. 结论

本项目在数据源层面采用“用户上传 + 公开数据库复现”的策略，兼顾教学易用性与科研可重复性。通过覆盖 PDB、GRO+XTC、PRMTOP+RST7 等主流组合，系统已具备较完整的拓扑-轨迹协同处理能力。后续可继续扩展 DCD、NetCDF 等轨迹格式，以提升跨软件生态互操作性。
